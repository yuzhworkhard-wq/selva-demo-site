# 生产统计页级联筛选设计

## 背景

当前生产统计页顶部只有“产品”单一下拉，个人统计和团队统计都直接复用这一个产品列表。项目数量较少时问题不明显，但当产品变多后，用户需要在“全部产品”的长列表里逐个寻找目标产品，定位成本很高。

数据中心页已经提供了更贴近真实使用习惯的三级筛选形式：`行业 -> 客户 -> 产品`。这次调整希望将生产统计页的产品筛选也升级为同样的逐级收缩模式，减少查找时间，并保持跨页面的交互一致性。

## 目标

- 在生产统计页顶部新增与数据中心一致的 `行业 -> 客户 -> 产品` 三联下拉。
- 每一级都保留“全部”选项，允许用户在任意层级停留，不强制逐级选满。
- 下一级选项根据上一级已选结果动态收缩，帮助用户快速定位目标产品。
- 个人统计和团队统计共用同一套顶部筛选入口，确保交互一致。
- 筛选结果同时作用于任务数、视频数、趋势图、任务分布和模型分布，保持统计口径统一。

## 非目标

- 不引入多选筛选弹窗。
- 不新增搜索框或模糊检索逻辑。
- 不修改数据中心现有交互。
- 不调整团队统计内部按成员、分类、业务的角色专属筛选逻辑，只是在其基础上叠加新的项目维度筛选。

## 现状

生产统计页的核心逻辑集中在 `render/pages.js`：

- `statsFilter` 目前只保存 `product`，没有 `industry` 和 `client`。
- `buildStatsFilterBar()` 只渲染项目属性、产品、快捷时间和时间范围。
- `filterStatsTasks()` 和 `filterProducedVideos()` 只对 `product` 做项目维度过滤。
- 团队统计的组长、经理、超管视图会在公共筛选栏后再附加成员/分类/业务筛选。

项目数据本身已经具备实现级联筛选所需字段，无需新增数据模型：

- `projects[].industry`
- `projects[].client`
- `projects[].product`

## 设计方案

### 顶部筛选结构

生产统计页顶部筛选栏新增两个下拉，最终排列为：

- 项目属性
- 行业
- 客户
- 产品
- 快捷时间
- 时间范围

视觉样式直接复用数据中心现有的标签加下拉形式，保持暗色背景、边框、圆角、字号和间距一致，不引入新的控件风格。

### 筛选状态

`statsFilter` 新增两个字段：

- `industry: 'all'`
- `client: 'all'`

保留现有：

- `product: 'all'`

这样统计页的项目维度筛选会变成三层状态组合，而不是单一产品值。

### 选项来源

新增一组基于“当前用户可见项目”生成的级联选项函数。

选项来源规则：

- 只从当前用户可见的 `projects` 中取值。
- 行业列表来自可见项目中的 `industry`。
- 客户列表来自满足当前行业筛选条件的项目中的 `client`。
- 产品列表来自满足当前行业和客户筛选条件的项目中的 `product`。
- 每组下拉都在最前面插入 `全部`。

这样能保证：

- 不会出现当前角色无权看到的数据项。
- 客户和产品不会出现与上级筛选条件无关的“脏选项”。

### 联动规则

#### 行业变更

当用户切换行业时：

- 客户列表立即按该行业收缩。
- 产品列表按新的行业和客户组合收缩。
- 如果当前客户值在新行业下不存在，自动重置为 `all`。
- 如果当前产品值在新的有效范围内不存在，自动重置为 `all`。

#### 客户变更

当用户切换客户时：

- 产品列表按新的行业和客户组合收缩。
- 如果当前产品值不在新客户范围内，自动重置为 `all`。

#### 产品变更

产品仍为最终层级，行为保持普通单选下拉。

### 数据过滤逻辑

统计页的公共项目维度过滤统一改为：

- 先按 `industry` 过滤
- 再按 `client` 过滤
- 再按 `product` 过滤
- 最后再叠加现有的项目属性、角色范围和时间范围过滤

适用对象：

- 任务统计
- 已产出视频统计

这保证以下模块口径一致：

- 总任务数
- 生成视频数
- 每日生产趋势
- 任务分布
- 视频模型分布
- 团队成员概览中的任务统计

### 与团队统计的配合

个人统计和团队统计继续共用同一个顶部公共筛选栏。

团队视图中原有的角色附加筛选保持不变：

- 组长视图继续使用“成员”下拉。
- 经理视图继续使用“分类”下拉。
- 超管视图继续使用“业务”下拉。

新的 `行业 / 客户 / 产品` 级联筛选是在这些角色范围过滤之后继续收口项目维度，不改变角色数据边界。

## 交互细节

- 初始状态显示：
  - 行业：全部
  - 客户：全部
  - 产品：全部
- 用户可以只选行业，不选客户和产品。
- 用户可以选行业后直接选“全部客户”下的某个产品，只要该产品在当前行业范围内存在。
- 当上一级切换导致下一级当前选项失效时，系统自动回退到“全部”，不弹提示，不中断操作。
- 筛选变化后立即刷新统计卡片和图表，保持当前页面的实时反馈方式。

## 实现边界

优先在 `render/pages.js` 内局部完成，不额外拆出新文件。原因是生产统计相关逻辑当前已经集中在同一处，实现一次性收口更安全，也更符合现有原型结构。

建议新增的最小辅助函数包括：

- 获取统计页可见项目集合
- 获取统计页级联选项
- 校验并修正无效筛选值
- 对任务和视频应用统一的项目维度过滤

## 测试策略

新增或补充 Node 内置测试，重点覆盖：

- 统计页行业选项只包含可见项目里的行业。
- 选中行业后，客户选项会收缩。
- 选中行业和客户后，产品选项会进一步收缩。
- 上一级切换后，如果下一级当前值失效，会自动回退到 `all`。
- `filterStatsTasks()` 和 `filterProducedVideos()` 会同时受 `industry / client / product` 影响。

## 风险与取舍

- 统计页当前没有现成的级联函数，新增辅助逻辑时要避免和数据中心逻辑出现两套相似但行为不一致的实现。可以借鉴数据中心的生成方式，但不强行抽成跨模块公共函数，先保证当前页面行为稳定。
- 当前统计逻辑里有一部分任务可能只依赖 `task.product` 而找不到明确项目记录。对于这类记录，本次筛选以项目维度为准：`industry` 和 `client` 过滤依赖项目信息，`product` 过滤继续兼容任务本身的产品字段。
- 当项目数据存在同名产品但属于不同客户时，三级级联能正确缩小范围，这也是本次设计的主要收益之一。
