# 团队层级与业务联动设计

## 背景

当前原型把“分组”建模成扁平小组，成员页、分组页、数据中心、统计页分别使用不同的推断逻辑：

- `data.js` 中的 `groups` 只表示普通小组。
- 数据中心在 `render/pages.js` 中通过 `media + 产品关键词` 推断业务。
- 团队成员页只展示首个分组标签，且所有具备团队管理权限的人都可改角色。

这和目标组织模型不一致。新需求要求把组织结构前置到人员管理中，并让数据中心筛选直接复用该结构。

## 目标

- 把团队结构升级为三级模型：一级业务、经理二级分类、成员归属。
- 保持“团队成员”列表对所有角色全量可见。
- 让编辑权限严格按管理链收口：
  - 超管可编辑所有成员，并且只有超管可修改角色身份。
  - 经理只能编辑自己负责一级业务内成员的归属。
  - 组长和成员没有成员编辑权限。
- 让数据中心和统计页基于成员归属来决定业务筛选与可见范围。

## 组织模型

### 一级业务

系统内固定存在 4 个一级业务：

- 出海互娱
- 本土互娱
- 出海工具
- 本土工具

一级业务只由超管维护。超管负责把人员归入一级业务，但不负责继续拆分经理以下的业务结构。

### 经理管理域

- 每个经理隶属于一个一级业务。
- 每个经理下固定存在两个二级分类：`a类`、`b类`。
- 经理默认管理自己所属一级业务下的成员，并且只可调整自己管理域内成员的二级分类归属。

### 组长与成员

- 组长和成员都挂在某个经理的 `a类` 或 `b类` 下。
- 组长不再创建组，也不再编辑下属资料或归属。
- “分组”在页面上展示为成员所属组织路径，例如：`出海互娱 / a类`。

## 数据建模

### 用户数据

`users` 从当前仅依赖 `groups` 的结构升级为显式组织字段：

- `businessId`: 一级业务 ID
- `managerScopeId`: 经理管理域 ID
- `subgroupKey`: `a` 或 `b`

同时保留兼容显示所需的基础字段，如 `role`、`email`、`status`。

### 组织定义

新增集中化组织配置，替代当前 `groups` 的扁平职责：

- `BUSINESS_OPTIONS`: 4 个一级业务
- `managerScopes`: 经理管理域列表，记录经理属于哪个一级业务
- 每个管理域默认派生 `a类` / `b类`

旧 `groups` 不再作为核心权限来源，可改造成由组织数据派生出的展示结构，避免大面积重写现有项目权限代码时失去兼容性。

## 页面行为

### 设置 -> 团队管理 -> 成员

- 所有人都能看到完整成员列表。
- “分组”列展示组织路径：
  - 超管：`出海互娱`
  - 经理：`出海互娱 / 经理`
  - 组长/成员：`出海互娱 / a类`
- 只有超管看到角色切换控件。
- 超管和经理可看到“编辑归属”入口：
  - 超管可编辑所有人。
  - 经理仅可编辑自己管理域内成员。
- 组长和成员没有任何编辑入口。

### 设置 -> 团队管理 -> 分组

“分组”页从扁平卡片改为组织结构视图：

- 超管视角：
  - 展示 4 个一级业务卡片
  - 每张卡片下展示该业务内经理、`a类` / `b类` 成员分布
  - 可进入详情并调整成员一级业务归属
- 经理视角：
  - 仅展示自己的一级业务
  - 展示并管理自己名下的 `a类` / `b类`
  - 只能调整自己管理域内成员的分类归属
- 组长/成员视角：
  - 只读展示，无创建或编辑入口

### 邀请成员

邀请弹窗需按角色收敛可配置项：

- 超管邀请：
  - 可设角色
  - 可指定一级业务
  - 若角色为组长/成员，可进一步指定经理与 `a类 / b类`
- 经理邀请：
  - 不可设为超管
  - 默认落到自己的一级业务
  - 只能指定 `a类 / b类`
- 组长/成员：
  - 不显示邀请入口

## 数据中心与统计页

### 可见范围

- 超管：看全部数据。
- 经理：只看自己一级业务内成员产出数据。
- 组长：只看自己下层成员数据。
- 成员：只看自己数据。

### 业务筛选

- 超管的数据中心“业务”下拉显示 4 个一级业务。
- 经理的数据中心“业务”下拉只显示 `a类`、`b类`。
- 组长和成员不显示业务下拉。

### 业务来源

视频记录不再主要依赖 `dcDeriveBusiness(project)` 的关键词推断，而是优先从上传人 `uploader` 的组织归属推导：

- 超管筛选一级业务时，匹配上传人的一级业务。
- 经理筛选 `a类 / b类` 时，匹配上传人在自己管理域下的分类。

项目维度的 `media / product / industry` 仍保留，用于原有非组织筛选。

## 权限规则

### 超管

- 查看全部成员
- 编辑全部成员资料与归属
- 修改全部成员角色
- 查看全部数据与 4 个一级业务

### 经理

- 查看全部成员
- 仅编辑自己管理域内成员归属
- 不可修改角色
- 在数据中心和统计页只看自己一级业务的数据
- “业务”筛选仅显示 `a类 / b类`

### 组长

- 查看全部成员
- 不可编辑成员资料与归属
- 仅看自己下层成员数据
- 不显示业务筛选

### 成员

- 查看全部成员
- 无成员管理能力
- 仅看自己数据

## 测试策略

- 新增 Node 内置测试入口，不依赖第三方包。
- 重点覆盖：
  - 组织路径展示
  - 成员编辑权限
  - 数据中心可选业务项
  - 数据中心按角色过滤数据范围

## 风险与取舍

- 现有项目权限大量依赖 `groups`。为了避免一次性重写过多页面，本次将优先新增组织辅助函数，并让旧 `groups` 作为兼容数据保留。
- 数据中心当前使用项目关键词推断业务，改成按上传人组织归属后，部分历史演示数据的业务标签会发生变化。这是预期内变更，因为新模型的目标就是“人员管理决定业务归属”。
